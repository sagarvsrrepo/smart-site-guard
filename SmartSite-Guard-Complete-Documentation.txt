SmartSite Guard - Complete Project Documentation
================================================

1. Project Overview
-------------------
SmartSite Guard is a fog-edge safety monitoring solution for construction sites.
It combines AWS IoT Core, a local fog processor, DynamoDB storage, and a Flask dashboard.
The system simulates on-site sensor data, processes it in a fog node, stores alerts and events in the cloud,
and provides a dashboard for filtered event visualization.

2. Architecture Summary
-----------------------
- Simulator publishes raw sensor readings to AWS IoT Core on topic `smartsite/raw`.
- Fog processor subscribes to raw sensor events, classifies them, stores processed events in DynamoDB,
  and publishes processed events to `smartsite/processed`.
- Dashboard reads events from DynamoDB and renders metrics, charts, alerts, and logs.

3. Key Components
-----------------
- `simulator/publisher.py`
  - Generates random sensor readings for temperature, gas, noise, vibration, and proximity.
  - Connects to AWS IoT Core using the default boto3 credential chain.
  - Publishes raw events to the IoT topic `smartsite/raw`.

- `fog/processor.py`
  - Subscribes to AWS IoT Core topic `smartsite/raw`.
  - Classifies sensor readings using `rules.classify_event`.
  - Stores processed events in DynamoDB table `SmartSiteGuardEvents`.
  - Publishes processed events to `smartsite/processed`.
  - Logs raw receive events and storage actions.

- `dashboard/app.py`
  - Flask web application.
  - Reads processed events from DynamoDB.
  - Supports quick time ranges and custom time range filters.
  - Provides sensor overview cards, alert list, event logs, and four charts.

- `cloud/template.yaml`
  - CloudFormation template to create the DynamoDB table only.
  - Uses a stack-specific table name to avoid collisions in shared environments.

- `cloud/deploy.ps1`
  - Deploys the CloudFormation stack using AWS CLI.
  - No IAM role creation required in Learner Lab.

5. Runtime Flow
---------------
Simulation and fog processing use AWS IoT Core and DynamoDB.

Simulator flow:
- Start `simulator/publisher.py`.
- AWS credentials are resolved from the default boto3 chain.
- The simulator discovers the IoT endpoint and connects.
- It publishes raw sensor messages every few seconds.

Fog flow:
- Start `fog/processor.py`.
- The fog node subscribes to `smartsite/raw`.
- Each raw message is received and logged.
- The fog node classifies the event and stores it in DynamoDB.
- It optionally publishes a processed message to `smartsite/processed`.

Dashboard flow:
- Start `dashboard/app.py`.
- User opens the browser at `http://127.0.0.1:5000`.
- The dashboard queries `/api/dashboard`.
- It applies quick time filters or custom start/end range filters.
- The dashboard updates graphs, alerts, and logs from the same data set.

6. Dashboard Filtering
----------------------
The dashboard supports:
- Last 5 minutes, 10 minutes, or 1 hour quick filters.
- Custom start/end time filters using datetime-local controls.
- Sensor type filtering.
- Severity filtering.

All dashboard panels refresh together so charts, summary, alerts, and logs stay in sync.

7. AWS Resources
----------------
The deployed AWS resources are intentionally minimal for Learner Lab:
- DynamoDB table `SmartSiteGuardEvents-<StackName>`
- AWS IoT Core topic names are configured in application environment variables.

8. Environment Variables
------------------------
Common environment variables:
- `AWS_DEFAULT_REGION` = `us-east-1`
- `DYNAMODB_TABLE` = `SmartSiteGuardEvents`
- `IOT_ENDPOINT` = `<your-iot-endpoint>.amazonaws.com`
- `IOT_TOPIC_RAW` = `smartsite/raw`
- `IOT_TOPIC_PROCESSED` = `smartsite/processed`
- `SITE_ID`, `ZONE_ID`, `PUBLISH_INTERVAL_SECONDS` for simulator customization.
